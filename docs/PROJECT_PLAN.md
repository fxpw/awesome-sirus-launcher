# План разработки Awesome Sirus Launcher

Документ описывает стадии разработки Electron-приложения для скачивания аддонов с GitHub, обновления клиента World of Warcraft Sirus, установки FPS-патча, бекапа папки `WTF` и запуска игры.

## 1. Цель

Сделать десктопный лаунчер, который:

- хранит путь к папке WoW;
- скачивает и обновляет аддоны из GitHub-репозиториев;
- в будущем умеет автообновлять WeakAuras;
- проверяет файлы клиента по MD5 и докачивает/обновляет отличающиеся файлы;
- ставит FPS-патч в `Data/ruRU/`;
- делает архивный бекап папки `WTF`;
- запускает игру;
- показывает понятный статус операций, прогресс, ошибки и журнал.

## 2. Рекомендуемый стек

Основной вариант:

- Desktop: `Electron`;
- сборка Electron: `electron-vite`;
- backend/main process: `TypeScript` на Node.js API внутри Electron;
- frontend/renderer: `Vue 3` + `TypeScript`;
- state: `Pinia`;
- UI: сначала свой минимальный дизайн на CSS/Tailwind, UI-kit добавлять только если он реально ускорит работу;
- упаковка: `electron-builder` или связка, которую рекомендует выбранный шаблон `electron-vite`;
- тесты: `Vitest` для модулей, `Playwright` для UI/e2e.
- CI/CD: GitHub Actions для проверки, сборки и будущего релизного autobuild.

Почему так:

- Electron дает зрелую экосистему для файловой системы, процессов, скачивания, автообновления и Windows-дистрибуции.
- `electron-vite` дает быстрый dev loop и поддерживает TypeScript, Vue и Electron preload/main/renderer из коробки.
- Vue 3 хорошо подходит под компактный лаунчер: настройки, списки аддонов, прогресс, модальные окна, журнал.

Альтернатива:

- `Tauri 2` + Vue 3.
- Плюсы: потенциально меньше вес и потребление памяти.
- Минусы: backend придется писать на Rust, файловые и сетевые операции надо проводить через Tauri commands/plugins, экосистема под game launcher сценарии обычно менее прямолинейна.

Решение на старте: использовать Electron + Vue 3 + TypeScript. Tauri рассмотреть позже, если размер приложения станет важнее скорости разработки.

Источники для выбора стека:

- Electron context isolation: https://electronjs.org/docs/latest/tutorial/context-isolation
- electron-vite: https://electron-vite.org/
- Tauri 2: https://v2.tauri.app/

## 3. Архитектура модулей

### Main process

Отвечает за системные операции:

- `app-paths` - хранение путей приложения, пользовательских данных, временной папки;
- `settings` - чтение/запись настроек, путь к WoW, выбранные источники, включенные аддоны;
- `wow-client` - проверка структуры клиента, поиск `Wow.exe`, `Data/ruRU`, `WTF`, запуск игры;
- `downloads` - очередь скачиваний, прогресс, retry, отмена, временные файлы;
- `patches` - получение списка файлов и MD5 из API, сравнение с локальными файлами, обновление клиента;
- `fps-patch` - скачивание `patch-ruRU-[.mpq` и установка в `Data/ruRU/`;
- `addons` - скачивание zip архива кода GitHub-репозитория, распаковка и установка папок аддонов в `Interface/AddOns`;
- `weakauras` - будущий модуль автообновления WeakAuras, источник и формат обновлений расписать позже;
- `backup` - архивирование `WTF`, список бекапов, восстановление;
- `launcher` - запуск игры с проверками;
- `logging` - файл логов и события для UI;
- `versioning` - чтение версии из `VERSION`/`package.json`, отображение версии в UI и синхронизация релизов.

### Preload

Единственная безопасная граница между UI и системным backend:

- экспортирует typed API через `contextBridge`;
- не отдает в renderer прямой доступ к Node.js;
- валидирует входные данные до IPC;
- возвращает DTO, а не внутренние классы.

### Renderer

Пользовательский интерфейс:

- `Dashboard` - общий статус клиента, быстрые действия;
- `Addons` - список аддонов, установка, обновление, удаление;
- `WeakAuras` - будущий экран автообновления WeakAuras, детали расписать позже;
- `ClientUpdate` - проверка MD5, список измененных файлов, прогресс;
- `FpsPatch` - статус патча, установка/переустановка;
- `Backups` - бекапы `WTF`, создание, восстановление, удаление;
- `Settings` - путь к WoW, источники API, директории, поведение запуска;
- `Logs` - читаемый журнал операций.

## 4. Источники данных

### MD5 API клиента

Использовать источники с fallback:

1. `https://s-patches.pro/api/client/patches`
2. `https://s-patches.ru/api/client/patches`
3. `https://sirus.world/api/client/patches`

Правила:

- пробовать источники по порядку;
- при ошибке сети или невалидном JSON переходить к следующему;
- кэшировать последний успешный ответ;
- сохранять timestamp проверки;
- не удалять локальные файлы без явного правила из манифеста;
- скачивать только отсутствующие или несовпадающие по MD5 файлы.

### FPS-патч

Основная ссылка:

- `https://d1st4r.ru/patch/patch-ruRU-[.mpq`

Альтернативная ссылка:

- `http://d1st4r.stream/patch/patch-ruRU-[.mpq`

Путь установки:

- `<wowPath>/Data/ruRU/patch-ruRU-[.mpq`

Правила:

- перед установкой проверить, что выбранный путь похож на папку WoW;
- создать `Data/ruRU`, если папки нет, только после подтверждения пользователем;
- скачивать во временный файл;
- после успешного скачивания атомарно перемещать файл в целевой путь;
- показывать размер, прогресс, источник и ошибку при fallback.

### GitHub-аддоны

В MVP аддоны скачивать не из releases/assets, а напрямую как zip архива кода GitHub-репозитория. После скачивания архив распаковывается во временную папку, верхняя папка GitHub-архива отбрасывается, а реальные папки аддонов переносятся в `<wowPath>/Interface/AddOns`.

На первой стадии поддержать конфиг со списком репозиториев:

```json
[
  {
    "id": "example-addon",
    "name": "Example Addon",
    "repo": "owner/repository",
    "ref": "master",
    "installDir": "ExampleAddon",
    "source": "source-zip"
  }
]
```

Правила установки:

- скачивать `https://github.com/{owner}/{repo}/archive/refs/heads/{ref}.zip` или другой zip URL для выбранного `ref`;
- всегда скачивать во временный файл;
- распаковывать во временную директорию;
- не копировать верхнюю папку вида `{repo}-{branch}` целиком в `AddOns`;
- искать внутри распакованного архива папки, где есть `.toc`;
- если `installDir` указан, ставить найденную папку под этим именем;
- если в архиве несколько папок аддонов, переносить каждую папку с `.toc` наверх в `Interface/AddOns`;
- перед заменой существующего аддона делать временный backup или atomic replace;
- защищаться от zip slip и путей за пределы временной директории.

Позже добавить:

- несколько папок внутри одного архива;
- pinned commit/tag;
- ручной zip URL;
- проверку установленной версии по `.toc`.

## 5. Стадии разработки

### Стадия 0. Подготовка проекта

Результат: запускаемый пустой Electron/Vue проект.

- создать проект на `electron-vite` + Vue + TypeScript;
- добавить `package.json` и синхронизировать `package.json.version` с `VERSION`;
- настроить ESLint, Prettier, Vitest;
- настроить базовые alias;
- завести структуру `main`, `preload`, `renderer`, `shared`;
- добавить CI-скрипты локальной проверки;
- подключить GitHub Actions build workflow;
- завести `CHANGELOG.md` и правило обязательного обновления changelog агентами;
- описать команды в `README.md`.

### Стадия 1. Каркас приложения

Результат: UI с навигацией и безопасным IPC.

- сделать главное окно;
- включить `contextIsolation`;
- отключить прямой Node.js доступ в renderer;
- сделать typed IPC контракт;
- добавить страницы Dashboard, Addons, Update, Backups, Settings, Logs;
- добавить общий store состояния операций.

### Стадия 2. Настройки и путь к WoW

Результат: приложение знает, где лежит игра.

- выбрать папку через системный dialog;
- проверить наличие `Wow.exe`, `Data`, `Interface`, `WTF`;
- сохранить путь в user data;
- показать статус валидности клиента;
- дать пользователю изменить путь;
- добавить тесты на валидацию путей.

### Стадия 3. Бекап папки WTF

Результат: безопасное создание и восстановление бекапов.

- архивировать `<wowPath>/WTF` в `.zip`;
- хранить бекапы в user data или выбранной папке;
- имя файла: `wtf-backup-YYYY-MM-DD-HH-mm-ss.zip`;
- показывать список бекапов;
- восстановление делать через временную папку и подтверждение;
- перед восстановлением создавать safety backup текущей `WTF`;
- добавить обработку больших папок и прогресс.

### Стадия 4. FPS-патч

Результат: установка патча в один клик.

- скачать патч по основной ссылке;
- при ошибке использовать альтернативную ссылку;
- сохранить в `<wowPath>/Data/ruRU/`;
- показывать текущий статус установлен/не установлен;
- добавить переустановку;
- логировать источник скачивания.

### Стадия 5. Проверка и обновление клиента по MD5

Результат: лаунчер умеет находить отличающиеся файлы клиента.

- получить манифест файлов из API;
- нормализовать пути под Windows;
- посчитать MD5 локальных файлов потоково;
- сравнить с манифестом;
- показать список отсутствующих/битых файлов;
- скачать нужные файлы во временную папку;
- проверить MD5 после скачивания;
- переместить файлы в клиент;
- поддержать retry и отмену.

Открытый вопрос: API может возвращать не только список MD5, но и URL/пути скачивания. Перед реализацией надо сохранить пример ответа и описать точную схему.

### Стадия 6. Аддоны с GitHub

Результат: установка и обновление аддонов.

- завести локальный каталог доступных аддонов;
- построить zip URL архива кода GitHub по `repo` и `ref`;
- скачать zip архива кода во временный файл;
- распаковать во временную папку;
- найти папки с `.toc`;
- отбросить верхнюю папку GitHub-архива и перенести папки аддонов наверх;
- установить в `<wowPath>/Interface/AddOns`;
- показывать установленную версию;
- добавить удаление аддона;
- добавить обновление всех аддонов.

### Стадия 7. Запуск игры

Результат: кнопка запуска WoW.

- определить исполняемый файл;
- запустить процесс через Electron main;

## 6. Версионирование, changelog и autobuild

Правила:

- `CHANGELOG.md` хранит все заметные изменения.
- Агент после изменения проекта обновляет `CHANGELOG.md` в разделе `[Unreleased]`.
- `VERSION` хранит текущую релизную версию.
- После появления `package.json` поле `package.json.version` должно совпадать с `VERSION`.
- Обычные изменения не обязаны повышать версию.
- Релиз начинается с отдельного изменения `VERSION` и переноса записей из `[Unreleased]` в новую версию.

GitHub Actions:

- PR должен запускать проверку, тесты и обычную сборку, когда появится Electron-проект.
- Push в `main`/`master` с изменением `VERSION` должен запускать релизную сборку.
- До появления `package.json` workflow должен пропускать Node/Electron шаги, чтобы плановая инфраструктура не ломала репозиторий.
- После появления `electron-builder` workflow должен сохранять Windows installer/portable как artifacts.

## 8. Риски

- У API патчей может быть нестабильная или недокументированная схема.
- HTTP fallback для FPS-патча не шифруется, его стоит использовать только как резервный источник.
- GitHub API имеет rate limits без токена.
- Антивирусы могут подозрительно относиться к лаунчерам, которые скачивают файлы и запускают `.exe`.
- Восстановление `WTF` может перезаписать настройки пользователя, поэтому нужен safety backup.
- Autobuild не будет выпускать полноценные артефакты, пока не появятся `package.json`, build script и packaging config.
