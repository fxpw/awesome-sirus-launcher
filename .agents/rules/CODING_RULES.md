# Правила разработки

Эти правила использовать при реализации лаунчера. Подробные project skills лежат в `.agents/skills/`.

## Общие правила

- После любых изменений проекта агент должен обновить `CHANGELOG.md` или явно указать, почему запись не нужна.
- При подготовке коммита агент должен использовать `.agents/skills/commit/SKILL.md` и описывать фактические изменения, а не только исходную просьбу.
- Версию релиза хранить в `VERSION`; после появления `package.json` синхронизировать `VERSION` и `package.json.version`.
- Писать на TypeScript везде, где это возможно.
- Держать типы общих DTO в `shared`, чтобы main/preload/renderer не расходились.
- Не смешивать UI, IPC и файловую систему в одном модуле.
- Любую операцию с файлами делать через main process.
- Renderer не должен иметь прямой доступ к Node.js API.
- Все пути хранить как абсолютные, но в логах и UI показывать аккуратно.
- Для путей использовать `path.join`, `path.resolve`, `path.normalize`, не собирать пути строками.
- Сетевые и файловые операции должны поддерживать отмену или хотя бы корректное завершение.
- Скачивание делать через временный файл, затем проверка, затем перемещение.
- Ошибки превращать в понятные доменные ошибки, а не отдавать в UI сырой stack trace.

## Backend / Electron main

- Включить `contextIsolation: true`.
- Не включать `nodeIntegration` в renderer.
- Все IPC-каналы описывать в одном typed контракте.
- IPC handlers должны валидировать входные данные.
- Не доверять данным из renderer.
- Долгие операции оформлять как jobs с `id`, `status`, `progress`, `error`.
- Прогресс отправлять событиями main -> renderer.
- MD5 считать потоково, без чтения больших файлов целиком в память.
- Архивы распаковывать только после проверки путей внутри архива, чтобы избежать zip slip.
- GitHub-аддоны в MVP скачивать как zip архива кода, распаковывать во временную папку и переносить в `Interface/AddOns` только реальные папки аддонов с `.toc`, а не верхнюю папку архива.
- Перед перезаписью важных пользовательских данных создавать safety backup.

## Frontend / Vue

- Компоненты делать маленькими и предметными.
- Бизнес-логику не класть в `.vue`, выносить в composables/stores.
- Для состояния использовать Pinia stores по доменам: settings, jobs, addons, backups, client.
- Не показывать технические исключения пользователю без перевода в понятный текст.
- Все кнопки опасных действий требуют подтверждения: восстановить `WTF`, удалить аддон, перезаписать файл.
- Прогресс должен быть видимым и не должен прыгать из-за изменения текста.
- UI должен быть плотным и рабочим: это лаунчер, не лендинг.
- Для иконок использовать библиотеку иконок после выбора UI-стека, не рисовать вручную.

## Тестирование

- Unit tests: чистые модули путей, MD5, парсинг API, выбор fallback-источника.
- Integration tests: backup/restore на временной директории.
- UI tests: старт приложения, выбор пути, отображение статуса.
- Для опасных файловых операций тесты должны использовать временные директории.
- Перед релизом обязательно прогнать smoke test на чистой папке клиента-копии.

## Версионирование и сборка

- Обычные изменения писать в `CHANGELOG.md` в раздел `[Unreleased]`.
- Релизную версию повышать только отдельным осознанным изменением `VERSION`.
- Когда появится `package.json`, его `version` должен совпадать с `VERSION`.
- GitHub Actions должен собирать приложение на PR и при изменении релизных файлов.
- Релизные артефакты Windows хранить как workflow artifacts, а публикацию в GitHub Releases добавить позже.

## Безопасность

- Никогда не выполнять скачанный код.
- Не запускать произвольный `.exe`, только найденный и подтвержденный `Wow.exe`.
- Не распаковывать архивы за пределы целевой директории.
- Не хранить токены в plain config, если позже появится GitHub token.
- HTTP-источники помечать как менее надежные.
- Любые destructive операции должны иметь dry-run или подтверждение.

## Стиль кода

- Предпочитать явные имена: `wowPath`, `addonsPath`, `backupArchivePath`.
- Не делать универсальные helper'ы до появления повторения.
- Не добавлять глобальное состояние без необходимости.
- Не ловить ошибку пустым `catch`.
- В логах фиксировать действие, путь, источник и результат.
- Комментарии писать только там, где они объясняют неочевидное решение.
